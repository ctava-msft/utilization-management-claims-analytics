# Specification — Production (Approach B)

> Personas, use cases, and acceptance criteria for the **Production deployment** of the UM Claims Analytics platform.
> Production replaces the manual VSCode/Copilot workflow with **AI agents** (Python FastAPI on AKS) that continuously generate UM insights, writing results to the Fabric Lakehouse for Power BI consumption.

---

## 1. Personas

| Persona | Role | How They Interact |
|---|---|---|
| **UM Analytics Agents (AI)** | Primary insight generators. Automated FastAPI services running on AKS that continuously read de-identified claims + policy data from the Lakehouse, perform CPT-code join analysis, call GPT-5.2-chat for summarisation, and write UM insights, flags, and alerts back to the Lakehouse. | Triggered by schedule, API call, or event → read Lakehouse → compute features → detect anomalies → call LLM → write results → expose via API. |
| **Agent Orchestrator** | Routes requests to the appropriate agent(s), manages multi-step workflows, and handles tool-calling dispatch across detection, policy simulation, appeals, and benchmarking agents. | Internal AKS service; not directly exposed to end users. |
| **UM Analyst (UPMC)** | Primary consumer. Reviews Power BI dashboards, investigates flagged anomalies, evaluates policy effectiveness, and provides feedback on detection accuracy. | Opens Power BI → explores UM dashboards → drills into flagged providers/suppliers → reviews LLM-generated explanations. |
| **UM Director (UPMC)** | Executive consumer. Reviews summary dashboards for strategic decisions — policy changes, resource allocation, fraud investigation prioritisation. | Reviews high-level Power BI dashboards → reads policy impact summaries → approves/rejects policy change recommendations. |
| **SIU Investigator (UPMC)** | Reviews fraud/waste/abuse flags generated by the agents. Prioritises cases for field investigation based on severity and evidence. | Filters Power BI dashboards by severity=high → reviews rule evidence + LLM explanation → exports case details for investigation. |
| **Solution Architect / Engineer** | Maintains and extends the agent pipeline — deploys new detection rules, updates prompt templates, tunes thresholds, monitors agent health. | Connects to VM via Bastion for admin access → deploys updates via ACR/AKS → monitors via Azure Monitor / Log Analytics. |
| **Data Engineer (UPMC)** | Maintains the Snowflake → Fabric data pipeline. Ensures de-identified claims data flows correctly and the Lakehouse schema stays current. | Manages Snowflake extracts → monitors OneLake shortcuts or ADF pipelines → validates Lakehouse table freshness. |

## 2. Use Cases

### UC-PROD-1: Automated Anomaly Detection & Summarisation

**Actor:** UM Analytics Agents (AI)

**Preconditions:**
- De-identified claims data is current in the Fabric Lakehouse.
- Structured policy JSON is present in the Lakehouse.
- Azure OpenAI deployment (GPT-5.2-chat) is active with `temperature=0` and fixed `seed`.

**Flow:**
1. Agent orchestrator triggers the detection pipeline (on schedule or via API).
2. Detection agent reads de-identified claims and policy data from the Lakehouse via Private Link.
3. Feature engineering computes provider, facility, and temporal metrics.
4. Detection rules flag outliers, OON DME clusters, billing anomalies, and new-entity volume spikes.
5. Flagged results (CPT-code joins + rule evidence) are sent to GPT-5.2-chat (`temperature=0`, fixed `seed`) for summarisation.
6. GPT returns plain-language anomaly explanations and flag recommendations.
7. Agent writes UM insights, flags, and alerts as Delta tables back to the Lakehouse.
8. Power BI dashboards refresh via DirectLake.

**Postconditions:**
- Lakehouse contains up-to-date UM insight, flag, and alert tables.
- Each flag includes: `rule_name`, `entity_id`, `feature_values`, `threshold`, `actual_value`, `severity`, `llm_summary`.
- Power BI dashboards reflect the latest run.

**Acceptance Criteria:**
- AC-PROD-1.1: End-to-end pipeline completes within SLA (< 15 min for 1M claims).
- AC-PROD-1.2: All detection rules from the POC are operational in Production.
- AC-PROD-1.3: LLM summaries are identical across re-runs with the same input data (determinism).
- AC-PROD-1.4: No PHI in any Lakehouse table, API response, or LLM prompt.

### UC-PROD-2: Interactive Dashboard Exploration

**Actor:** UM Analyst / UM Director / SIU Investigator

**Preconditions:**
- Agents have completed at least one pipeline run (UC-PROD-1).
- Power BI is connected to the Lakehouse via DirectLake.

**Flow:**
1. User opens Power BI and navigates to the UM Analytics workspace.
2. Summary dashboard shows key metrics: total flags by severity, denial rate trends, top flagged providers, policy impact summaries.
3. User drills into a specific flag → sees rule evidence (rule name, feature values, threshold, actual value) alongside the LLM-generated plain-language explanation.
4. User filters by time period, service category, geography, or severity.
5. User exports a subset for offline review or shares with colleagues.

**Postconditions:**
- User has reviewed current UM analytics and identified items for action.

**Acceptance Criteria:**
- AC-PROD-2.1: Dashboard loads within 5 seconds (DirectLake performance).
- AC-PROD-2.2: Every flag row displays both structured evidence and LLM summary.
- AC-PROD-2.3: Filters for severity, time, geography, and service category are functional.
- AC-PROD-2.4: Data is current to within the agent pipeline SLA.

### UC-PROD-3: Policy Impact Analysis

**Actor:** UM Analytics Agents (AI) + UM Director

**Preconditions:**
- Claims data spans a policy change event (e.g., authorization requirement removed for a service category).

**Flow:**
1. Policy simulation agent reads claims data before and after the effective date.
2. Computes pre/post metrics: volume, cost, denial rate, OON rate for affected services.
3. Detects rebound patterns (utilisation returning to pre-change levels).
4. Sends results to GPT-5.2-chat for plain-language impact summary.
5. Writes policy impact table and LLM summary to the Lakehouse.
6. UM director reviews the policy impact dashboard in Power BI.

**Postconditions:**
- Policy impact table with effect sizes is available in Power BI.
- LLM summary explains direction, magnitude, and any rebound.

**Acceptance Criteria:**
- AC-PROD-3.1: Pre/post comparison table shows volume and cost deltas.
- AC-PROD-3.2: Rebound detection identifies services returning to pre-change levels.
- AC-PROD-3.3: Effect sizes are clearly labelled with confidence indicators.

### UC-PROD-4: Appeals Funnel Analysis

**Actor:** UM Analytics Agents (AI) + UM Analyst

**Preconditions:**
- Claims data includes denial and appeal fields.

**Flow:**
1. Appeals agent computes denial→appeal conversion rates by denial_reason_category.
2. Identifies top denial categories by volume and appeal rate.
3. Estimates administrative cost proxy (appeals count × configurable cost-per-appeal).
4. Identifies providers or third-party patterns with disproportionate appeal volumes.
5. Writes results to the Lakehouse.

**Postconditions:**
- Appeals funnel table and admin cost estimates available in Power BI.

**Acceptance Criteria:**
- AC-PROD-4.1: Denial→appeal funnel shows conversion rate by category.
- AC-PROD-4.2: Top 5 categories by appeal volume are highlighted.
- AC-PROD-4.3: Estimated admin cost is computed and displayed.

### UC-PROD-5: API-Driven Flag Retrieval

**Actor:** External systems / SIU Investigator

**Preconditions:**
- Agents have written flags to the Lakehouse.
- FastAPI endpoints are exposed via F5.

**Flow:**
1. Consumer calls the flags API endpoint (authenticated via Entra ID).
2. API returns a paginated list of flags with structured evidence and LLM summaries.
3. Consumer filters by severity, date range, entity type, or rule name.

**Postconditions:**
- Flags are delivered as structured JSON via REST API.

**Acceptance Criteria:**
- AC-PROD-5.1: API responds within 2 seconds for paginated queries.
- AC-PROD-5.2: Response includes `rule_name`, `entity_id`, `feature_values`, `threshold`, `actual_value`, `severity`, `llm_summary`.
- AC-PROD-5.3: Authentication is enforced; unauthenticated requests return 401.
- AC-PROD-5.4: No PHI in any API response.

### UC-PROD-6: Deploy Updated Detection Rules

**Actor:** Solution Engineer

**Preconditions:**
- A new detection rule or threshold change has been developed and tested.

**Flow:**
1. Solution engineer pushes a new container image to ACR with updated rules.
2. AKS rolling deployment updates the agent pods.
3. Solution engineer triggers a pipeline run to validate the new rules.
4. Reviews output in Power BI and compares to the prior run.

**Postconditions:**
- Updated rules are live in Production.
- New flags appear in subsequent pipeline runs.

**Acceptance Criteria:**
- AC-PROD-6.1: Rolling deployment completes with zero downtime.
- AC-PROD-6.2: New detection rules produce expected flags on test data.
- AC-PROD-6.3: All existing tests pass in CI before deployment.

### UC-PROD-7: Monitor Agent Health

**Actor:** Solution Architect

**Preconditions:**
- Azure Monitor / Log Analytics Workspace is configured.

**Flow:**

2. Reviews agent pod health, pipeline run durations, error rates, and LLM token usage.

## 3. Data Privacy Constraints (Production)

| Constraint | Enforcement |
|---|---|
| **No PHI in claims data** | De-identification is applied in Snowflake before extraction. The Snowflake team owns this boundary. |
| **No PHI in LLM prompts** | Agents send only de-identified CPT-code join results and policy text to Azure OpenAI via Private Endpoint. |
| **No PHI in Lakehouse** | All tables (claims, policy, UM outputs) contain only de-identified data. |
| **No PHI in API responses** | FastAPI endpoints return only de-identified flag data. |
| **No PHI in Power BI** | Dashboards consume Lakehouse tables that contain no PHI. |
| **No PHI in logs** | Agent logs do not include claim-level data; only aggregate counts, error messages, and run metadata. |

## 4. LLM Governance (Production)

| Parameter | Value | Rationale |
|---|---|---|
| **Model** | GPT-5.2-chat (Azure OpenAI, Private Endpoint) | Customer-approved model; accessed only over private network. |
| **`temperature`** | `0` | Eliminates sampling randomness; ensures deterministic completions. |
| **`seed`** | Fixed integer (configurable per deployment) | Combined with `temperature=0`, maximises output reproducibility. |
| **`max_tokens`** | Capped per prompt type | Prevents runaway token usage; tuned per agent. |
| **Prompt templates** | Version-controlled in the repository; deployed as part of the container image. | Ensures prompt changes go through code review and CI. |
| **Output grounding** | LLM summaries augment rule-based flags; they never replace structured evidence (rule name, feature values, threshold). | Every flag is explainable without the LLM. The LLM adds natural-language context for human consumption. |
| **Token usage tracking** | Logged per run in Azure Monitor. | Enables cost accounting and anomaly alerting on token consumption. |

## 5. Agent-to-Human Handoff

The agents are **insight generators**, not decision-makers. The following handoff points ensure human oversight:

| Stage | Agent Responsibility | Human Responsibility |
|---|---|---|
| **Flag generation** | Detect anomalies, apply rules, generate LLM summaries, write to Lakehouse. | UM analyst reviews flags, validates relevance, decides on action. |
| **Policy impact** | Compute pre/post metrics, detect rebounds, summarise impact. | UM director reviews impact, decides whether to keep/modify/revert policy. |
| **Fraud investigation** | Flag suspicious patterns with evidence and severity ranking. | SIU investigator reviews flags, prioritises cases, conducts field investigation. |
| **Threshold tuning** | Report false-positive/negative rates. | Solution engineer adjusts thresholds based on analyst feedback. |

## 7. Transition from POC

| POC Artefact | Production Equivalent |
|---|---|
| VSCode + Copilot workflow on VM | Python FastAPI agents on AKS |
| Manual `um-claims run-all` | Scheduled / API-triggered agent pipeline |
| Markdown reports on VM filesystem | Power BI dashboards via DirectLake |
| JSON flag files on VM | Lakehouse Delta tables + REST API |
| Solution architect reviews output | UM analyst / SIU investigator review dashboards |
| Single-run, manual seed | Continuous runs, fixed seed for reproducibility |
| VM-only compute | AKS horizontal pod autoscaling |
| No load balancer | F5 TLS termination + load balancing |
| Direct Azure OpenAI access | Private Endpoint access only |

> The core analytics logic (feature engineering, detection rules, policy simulation, appeals analysis, benchmarking) is **identical** between POC and Production. Production wraps it in a FastAPI service, adds orchestration, and replaces the manual VSCode/Copilot-assisted workflow with automated agent execution.
